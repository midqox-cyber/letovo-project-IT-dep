# Слой интеграции (SSO / внешний портал)

Этот документ описывает, **как подключить внешний портал** (единый вход и/или данные),
не ломая работающий сайт. Главный принцип:

> **Интеграция — это надстройка, а не замена.**
> Сайт работает на собственном входе и данных. Портал подключается отдельным слоем
> за флагом, с откатом (fallback) на локальный режим, если портал недоступен.

Пока флаг `GREENNET_AUTH_MODE=local` — всё работает как сейчас, ничего не меняется.

---

## 0. РЕАЛИЗОВАНО: вход через портал «Летово Игра» (`letovo-dev/letovo-all`)

Портал — сервер на C++ (PostgreSQL, REST + WebSocket) с **JWT/Bearer**-аутентификацией.
У него уже есть пользователи, роли, департаменты и валюта. Мы — **потребитель его API**,
всё **server-to-server** (у портала нет CORS, из браузера напрямую нельзя).

**Как включить (когда будет боевой адрес портала):**
```bash
GREENNET_AUTH_MODE=portal
GREENNET_PORTAL_URL=https://<адрес-портала>      # база API
# опц. маппинг, если их роли/департаменты называются иначе:
GREENNET_PORTAL_ROLE_MAP='{"teacher":"dept_head","student":"observer"}'
GREENNET_PORTAL_DEPT_MAP='{"1":"Департамент управления","5":"Научный департамент"}'
```

**Поток входа** (`POST /api/auth/portal`, реализовано и протестировано на моке):
1. Участник логинится на портале → получает Bearer-JWT.
2. Наш сервер валидирует токен: `GET {PORTAL}/auth/amiauthed` с `Authorization: Bearer`.
3. Логин берём из payload JWT (без проверки подписи — её проверил портал).
4. Профиль и роли: `GET /user/full/:username`, `GET /user/roles/:username`.
5. Маппим роль/департамент → `upsert_external_user` → открываем нашу сессию.
   Новому участнику ставится маппед-роль; существующему роль **не меняем** (её ведёт
   админ — защита от эскалации). Портал недоступен → 503, работает свой вход.

Фронтенд: на странице входа появляется кнопка **«Войти через портал»** (или авто-вход
по `?portal_token=...`, когда сайт встроен в портал). Свой вход остаётся как fallback.

**Осталось подтвердить у команды портала:** боевой base URL; точные поля ответов
`/user/full` и `/user/roles` (в их доке тела ответов пустые — мы читаем частые имена
`name/department/roles`, подстроим по факту); соответствие их `department`-id → наши 8;
разрешение обращаться к API и лимиты. Валюта: их `/transactions/*` (единая экономика)
или наши энеоины — на выбор (отдельный шаг).

## 2. Единый вход (SSO) через OpenID Connect

Когда у портала появится OAuth2/OIDC, вход добавляется **тремя шагами**, не трогая
остальной код:

1. **Кнопка** на странице входа (если `auth_mode == "sso"`) → редирект на портал.
2. **Callback-маршрут** принимает код, меняет его на токен, читает профиль пользователя.
3. Вызывает `upsert_external_user(...)` и открывает сессию — дальше всё как обычно.

Набросок маршрута (добавить в `app.py`, когда будут реквизиты провайдера):

```python
# pip install authlib
@app.get("/api/auth/sso/callback")
def sso_callback():
    if AUTH_MODE != "sso":
        return jsonify(error="SSO выключен"), 404
    token = oauth.portal.authorize_access_token()      # проверка на сервере
    profile = token["userinfo"]                          # {sub, name, email, ...}
    db = get_db()
    row = upsert_external_user(
        db,
        ext_username=profile["preferred_username"],
        name=profile.get("name"),
        # роль НЕ берём с портала автоматически — по умолчанию observer,
        # либо сопоставляем по таблице ниже, если портал ей доверяет
    )
    session.clear(); session["uid"] = row["id"]; session.permanent = True
    return redirect("/")
```

> **Важно:** секрет клиента (`client_secret`) и токены живут **только на сервере**.
> В браузер они не попадают никогда.

### Сопоставление ролей (портал → наши)

Роли у портала свои. Заполнить, когда будут известны:

| Роль портала | Наша роль |
|---|---|
| (например) `teacher` | `hq_head` |
| `student` | `observer` |
| … | … |

По умолчанию новый внешний участник получает `observer` — **повышает только админ**
(это принципиальная защита от самоназначения прав).

## 3. Данные с портала (новости, расписание, карта)

Если портал отдаёт данные по API — подтягиваем их **через наш сервер-прокси**
(браузер к чужому API напрямую не ходит из-за CORS), с кэшем и откатом:

```python
import urllib.request, json
def portal_get(path):
    if not PORTAL_BASE_URL:
        return None                                  # портал не настроен → fallback
    req = urllib.request.Request(PORTAL_BASE_URL + path,
                                 headers={"Authorization": f"Bearer {PORTAL_API_KEY}"})
    try:
        with urllib.request.urlopen(req, timeout=5) as r:
            return json.loads(r.read())
    except Exception:
        return None                                  # портал недоступен → показываем локальное
```

Адаптер приводит формат портала к нашему (см. поля в `app.py`: `news_public`,
`sched_public`, `obj_public`) и кладёт в базу или отдаёт напрямую. Если `portal_get`
вернул `None` — используем локальные данные. Так портал **не может «положить» сайт**.

## 4. Риски и как они закрыты

| Риск | Мера |
|---|---|
| Нет API / нет разрешения портала | Интеграция за флагом; без неё сайт полностью рабочий |
| Данные детей (10–17 лет), 152-ФЗ | Подключать только с разрешения организаторов; согласие фиксируется (`terms_at`) |
| Утечка секретов | `client_secret`/ключи только на сервере, в `.gitignore`, не в браузере |
| Портал изменил API / недоступен | Кэш + fallback на локальные данные, таймауты |
| CORS | Ходим через backend-прокси, не из браузера |
| Самоназначение прав через SSO | Новый внешний юзер = `observer`; роль меняет только админ |
| Регрессии при подключении | Флаг + тесты (58 шт.) + отдельная ветка/staging |

## 5. Что нужно от портала, чтобы включить SSO

- [ ] Протокол: OAuth2 / OpenID Connect (желательно) или SAML
- [ ] `client_id` и `client_secret` для нашего приложения
- [ ] Разрешённый `redirect_uri` (адрес нашего callback)
- [ ] Эндпоинты: authorization, token, userinfo (или OIDC discovery URL)
- [ ] Какие поля возвращает `userinfo` (логин, имя, роль, департамент)
- [ ] Официальное разрешение организаторов на доступ к данным участников

Когда это есть — включаем `GREENNET_AUTH_MODE=sso`, добавляем маршрут из §2, и всё.
